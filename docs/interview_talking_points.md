# Interview Talking Points

## 30-Second Project Explanation

This project is a ROS 2 Physical AI task executor. It takes a natural language command, converts it into a validated typed task plan, executes that plan through a C++ BehaviorTree.CPP executor, and routes robot primitives through a MoveIt-ready `/robot_action` service. The current system uses mock robot actions by default, with optional OpenAI planner mode and an optional MoveIt 2 compile path prepared for future robot-specific integration.

## 2-Minute Technical Explanation

The architecture separates language planning from physical execution. A Python planner node subscribes to `/user_command`, runs either a mock planner or optional OpenAI planner, and validates the resulting dictionary against an allowed task schema. Only supported task types and whitelisted steps can become a typed `pai_task_msgs/msg/TaskPlan`.

The C++ BehaviorTree.CPP executor subscribes to `/task_plan`, loads the `pick_and_place` behavior tree XML, and maps each tree node to a `/robot_action` service request. The robot action service is implemented in C++ and routes requests through `MotionPlanner` and `GripperController` abstractions. Those classes run in mock fallback mode by default and are structured for future MoveIt 2 `MoveGroupInterface` integration.

This gives the project clear safety boundaries: LLMs can propose high-level plans, ROS 2 messages define typed contracts, BehaviorTree.CPP handles deterministic execution, and robot actions are isolated behind a service layer.

## Why This Is Physical AI

It connects language-level intent to robotics execution while respecting physical-system constraints. The project is not just a chatbot; it includes ROS 2 interfaces, typed task messages, BehaviorTree.CPP execution, robot action services, and a MoveIt-ready motion abstraction.

## Why BehaviorTree.CPP Was Used

BehaviorTree.CPP is a good fit for robotics tasks because it provides explicit control flow, reusable action nodes, deterministic execution semantics, and clear failure handling. It is easier to inspect and extend than a free-form script generated by an LLM.

## Why The LLM Does Not Directly Control The Robot

Direct LLM control is unsafe because models can hallucinate actions, skip important steps, or produce commands outside the robot's capability set. This project treats LLM output as an untrusted planning suggestion. The schema validator rejects invalid task types or unsupported steps before anything reaches execution.

## Why MoveIt 2 Is Behind An Optional Build Flag

MoveIt 2 is robot- and environment-dependent. Keeping it behind `BUILD_WITH_MOVEIT=ON` lets the default project build in mock-only environments while still preparing a clean integration path for machines that have MoveIt 2 installed.

## What I Would Improve Next

- Add a real robot model with URDF, SRDF, controllers, and RViz configuration.
- Replace placeholder MoveIt paths with real `MoveGroupInterface` planning calls.
- Add perception to populate object poses dynamically.
- Add retry and recovery behavior in BehaviorTree.CPP.
- Add CI using a ROS 2 Docker image.
- Add unit tests for planner validation and service routing.

## Current Limitations

- The project does not yet control a real robot.
- MoveIt 2 execution is prepared but not fully configured.
- Object locations are symbolic placeholders rather than perception-derived poses.
- The current behavior tree covers a narrow pick-and-place demo path.
- The OpenAI planner is optional and requires local environment configuration.
